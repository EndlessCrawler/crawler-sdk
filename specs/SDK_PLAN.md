# crawler-sdk — SDK Refactor Plan

**Status:** _Implementation underway — **P1–P8 executed**._ The refactor rewrote the SDK API (functional core + thin wrapper, schemas, per-world exports, documented surface). It ran **after** the V2 modernization (`specs/V2_PLAN.md`) landed the stack work, but **before anything is published to npm** — the first publish happens only once the refactor is done (P10).

**What this document is.** The plan of record for the refactor: the **implementation steps** (below — executed steps first, remaining steps after) and the **open decisions still in flux**. Everything **settled** lives in **`specs/SDK_SPECS.md`** (the single home of every settled fact); this plan only *points* there (`→ SPECS §Section`) — it never restates specification. Code and SPECS move in lockstep (see `CLAUDE.md`).

**Working model.** The user throws ideas and specs (via the **`/sdk-plan`** project command — `.claude/commands/sdk-plan.md`); Claude fits them into this doc, keeps it organized, and **asks when an idea conflicts with the current implementation or is internally inconsistent** rather than silently reconciling it. The doc states **current facts only** — no dates or changelog narration; when a decision settles its final form is written into SPECS and its entry here collapses to a pointer, and when it changes the entry is rewritten to the outcome (git history carries the chronology). All state lives in this file and SPECS — a fresh clone + `/sdk-plan` resumes the work seamlessly.

**No back-compatibility constraint.** Nothing is published yet, so there are no external consumers to preserve. The only in-repo consumers (`crawler-react`, `apps/sdk-explorer`) and the sibling `ec-dapp` all link the workspace and move in lockstep — break APIs freely, fix consumers in the same change.

## Implementation phases

Build order: consumption-first — types, then the core that reads them, then the pipeline that produces data, then the apps; **C&C last**, exercising the whole chain a second time. TSDoc is definition-of-done in every phase. **P1–P8 are executed;** the remaining work is the ec-dapp import against a local link (P9), then the first npm publish and re-import from npm (P10), then C&C (P11).

- **P1 — Types & schemas. ✅ DONE.** `bigintish/`, the `DataSchema` descriptors + derived types, `ChamberData<Schema>` + `Door`, `World`/`View` types, chains rework, no `any` anywhere in core, `luw` deleted. → SPECS §Schemas, §`BigIntish`, §Worlds & Views.
- **P2 — Core / client. ✅ DONE.** Global store + `Options` bag + module/namespace ceremony + DOM events deleted (the `./internal` subpath gone); functional read surface (`loadWorld`, pure per-view reads, pure merge); the NEWS library (`coords/news`, ec fixtures pass unchanged) + name→library registry; `createCrawler` + `Crawler` + `WorldHandle` + `Chamber` + coarse subscription + `Converter` interface. A one-off migration script rewrote mainnet + goerli to the World shape (door `destCoord`/`destTile` derived, validated by `loadWorld`), seeding the P5 converter's derivations. Consumers rode keep-lights-on passes (api rewritten at P3, react at P7, explorer at P8). → SPECS §The `Crawler` client, §Worlds & Views.
- **P3 — api contract layer. ✅ DONE.** The typed contract surface: ABI codegen (artifact JSON → git-ignored const-asserted TS, regenerated before every build/typecheck/test), `getWorldContract` + cards/ERC-20/ERC-721/typed factories, parsed-result helpers (`readTokenMetadata`/`readTotalSupply`/`readOwnerOf`), warned public-RPC fallback, `BigIntish` boundaries; the global RPC registry + untyped read path deleted (`formatViewData` stayed put). → SPECS §`crawler-api`, §Chains, §Canonical serialization.
- **P4 — EC cache. ✅ DONE.** The single private `cache` package: one generic `fetch:tokens` script archives every token's `tokenURI` via P3 — block-pinned, missing-only, RPC-required, ≤ 10 req/s — into a `json`+`svg` (+`html` for `ec`) triple per token + a per-`dataDir` `_cache.json`; `ec` worlds embed the on-chain `Crawl.ChamberData` struct as `chamber` (the SVG alone doesn't carry the full map data). Mainnet landed at **326 tokens** (goerli unfetchable/frozen). Invalidation landed as an empty schema-level core policy, filled at P7. → SPECS §Data pipeline.
- **P5 — EC converter. ✅ DONE.** `EcTokenPayload` → `ChamberData<ec>`: pure, sync, struct-fed — **no SVG parsing** (map facts incl. `seed` ride in the embedded `chamber` struct; the SVG passes through display-only); readable facts normalized from the traits; the migration's derivations reproduced and pinned by the equivalence suite over the full cache. → SPECS §Data pipeline, §`ChamberData<Schema>`, §`Door`.
- **P6 — EC world data (builder + per-world exports). ✅ DONE.** The builder re-emits mainnet from the cache via the canonical serializer — fully offline, deterministic (byte-identical modulo the re-stamped `timestamp`), `worldInfo` re-stamped, first `TokenSvg` view; goerli stays frozen as migrated (still shipped). `crawler-data` moved to per-world subpath exports, each default-exporting a `WorldBundle`; the root ships no world JSON. → SPECS §Data pipeline, §Package map, §The `Crawler` client (`WorldBundle`).
- **P7 — react (hook surface + live path). ✅ DONE.** The full react surface + the cross-package live path: base-hook + alias model (`useWorldSchema`/`useChamberSchema` → `useWorldEC`/`useChamberEC` + union forms; world name optional everywhere) plus enumeration/info/selector/neighbors/`useTokenSvg` + `<ChamberSvg>`; `watchMints` + schema-dispatched payload assemblers in `crawler-api` (consumed by the cache — one fetch/assembly implementation), caller-supplied viem clients api-wide, the zero-config `liveUpdate` prop + localStorage persistence; the invalidation primitive in core (`getInvalidatedCoords`) + the cache staleness pass, verified against live mainnet. → SPECS §`crawler-react`, §Schemas (invalidation), §`crawler-api`.
- **P8 — sdk-explorer. ✅ DONE.** Rebuilt on the new API: the **full browse tool** (world index → SVG-grid chamber index → chamber detail with door navigation, separator-less route slugs, goerli data-only behind `hasView`) on the P7 react surface + the **same-origin data API** (the `/api/data` family, the converted `/api/onchain` route, `EXPLORER_CORS_ORIGINS` opt-in); the raw `/api/read` scaffold deleted; verified live. → SPECS §`apps/sdk-explorer`.
- **P9 — Import into Endless Crawler (ec-dapp), local path.** Point ec-dapp at the workspace packages via **local path** (not npm) and migrate it onto the new SDK — **validating the API, fixing bugs, and improving it against a real game before anything is published**. Owner helpers + delegate.xyz land with the game's needs (#17); **migrate ec-dapp's original→playable SVG converter into the SDK**. _Gate: #17._
- **P10 — Publish.** Surface **frozen** (→ SPECS §Public surface); TSDoc complete + **`apps/docs` stands** (hybrid, gated by `check:docs`); README/CLAUDE.md/PUBLISH.md updated. The **first npm publish** — run by the user per `PUBLISH.md`. Then **re-point ec-dapp from the local path to the published npm packages** (unblocking its V2 Phase 9).
- **P11 — C&C cache & world building.** The C&C world added to the `cache` package (`dataDir: cryptsandcaverns`); the `cnc` schema, converter, and world build — the multi-schema seam exercised end-to-end. _Gate: #14._

## Open specs & decisions

Numbers are **stable identifiers**, not an order. Each stays here until it settles, at which point its final form moves into SPECS and it drops from this list.

### For P9 — Endless Crawler import

- **#17 — Player-owned chambers: ownership resolution.** _(Requirement settled — owner helpers served from `crawler-api`, scoped to the connected player; delegate.xyz support required; mechanism OPEN.)_ Sub-questions:
  - **RPC-frugal lookup:** resolve the connected player's tokens **without indexers and without `ERC721Enumerable` enumeration** (one call per token). Candidates to evaluate: `Transfer`-event log scans (`eth_getLogs` on the two `Transfer(from|to=player)` filters, cached by block range), multicall batching, or accepting enumeration for small collections. Unresolved.
  - **View or not:** is ownership a world view (`TokenOwner`) at all? It's dynamic (transfer, delegation), so a static view is stale by nature — it may be runtime-api-only, optionally localStorage-cached like live chambers. _Naming lean: `TokenOwner` (consistent with `TokenCoord`)._
  - **delegate.xyz:** a connected player can use tokens stored in a different wallet. **Not yet investigated — research first** (registry lookup API, RPC cost, chain coverage) before designing this decision.

### For P11 — C&C

- **#14 — `cnc` coordinate mapping.** Crypts & Caverns dungeons have **no native coordinates**, but `ChamberData` is keyed by coord and gameplay navigates by neighbor-coord math. Interim rule: **`coord = chamber ID`**; the `cnc` importer must map/derive real coordinates. How they're calculated is unspecced — capture the spec when it arrives. **A v1 blocker:** the `cnc` converter ships in v1 and can't emit coord-keyed `ChamberData` without this.

### Cross-cutting / future

- **Cross-world doors.** A chamber in world A will eventually connect to a chamber in world B. The **`Door` element is where the connection lives**: `destCoord` today; a cross-world door widens the destination to a **world-qualified** form (`{ world, coord }`), with a same-world neighbor as the degenerate case. **OPEN:** how the destination world is identified in stored data. The `Crawler` container already owns cross-world traversal (→ SPECS §The `Crawler` client).
